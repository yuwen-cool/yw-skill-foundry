Use this schema when `evidence.py create` builds a
`yw-skill-foundry.run-compare/v2` manifest.

## Namespace compatibility

New manifests and reports use the `yw-skill-foundry.*` namespace:
`yw-skill-foundry.run-compare/v2`, `yw-skill-foundry.runs/v1`,
`yw-skill-foundry.judges/v1`, `yw-skill-foundry.contract/v1`,
`yw-skill-foundry.contract-result/v2`, and
`yw-skill-foundry.routing-result/v1`.

Verification remains backward-compatible with historical
`skill-foundry.*` and `skill-craft.*` v1/v2 manifests and nested artifacts.
Compatibility is verification-only: newly created artifacts never emit either
legacy namespace.

## Required files

- `task.md`, `judge-prompt.md`, `mapping.json`, `verdict.md`
- `runs.json`, `judges.json`
- every output, contract input/report, blind candidate, and judge evidence file
  referenced by those documents
- `skill-snapshot.md` and `manifest.json` are created by `evidence.py`

All referenced paths are relative, remain inside the evidence directory, and
resolve to non-empty regular files. The manifest hashes every required and
referenced artifact.

## `runs.json`

```json
{
  "schema": "yw-skill-foundry.runs/v1",
  "evaluation_level": "production",
  "metrics_status": "unavailable",
  "metrics_unavailable_reason": "Host does not expose token/time metrics.",
  "runs": [
    {
      "id": "run-a1",
      "configuration": "baseline",
      "run_number": 1,
      "model": "declared-model-label",
      "output": "baseline-1.md",
      "contract_file": "contract.json",
      "manual_results_file": "manual-baseline-1.json",
      "contract_report": "contract-baseline-1.json",
      "contract_passed": false
    }
  ]
}
```

Use `metrics_status: "recorded"` only when every run includes non-negative
`tokens` and `duration_seconds`. Otherwise give a concrete unavailable reason.

Production requires at least two matched run numbers per configuration. Run
IDs, output paths, and contract report paths are unique. Baseline and
with-skill use the same declared generator model label.

`contract_report` must be generated by `contract_eval.py` schema v2. It binds
the contract, output, and manual-results files by relative path and SHA-256,
covers every contract criterion exactly once, and has no pending manual check.

When a generated artifact includes a routing fixture, add all five fields:
`routing_fixture`, `routing_judgments`, `routing_worksheet`, `routing_report`,
and `routing_passed`. Create the report with:

```text
trigger_eval.py score --fixture ... --judgments ... --worksheet ... --report ...
```

The routing report binds all three inputs by relative path + SHA-256.
`evidence.py verify` recomputes phase/family scores and pass state; a copied or
hand-edited green report fails verification.

## `judges.json`

```json
{
  "schema": "yw-skill-foundry.judges/v1",
  "judges": [
    {
      "id": "judge-a1",
      "model": "declared-judge-model-label",
      "run_number": 1,
      "candidate_a": "baseline",
      "candidate_b": "with_skill",
      "candidate_a_file": "r1-a.md",
      "candidate_b_file": "r1-b.md",
      "prompt_file": "judge-prompt.md",
      "evidence_file": "judge-r1-normal.md",
      "verdict": "with_skill"
    }
  ]
}
```

Candidate files must be byte-identical to the output for their logical
configuration and `run_number`. Judge IDs and evidence paths are unique.
Verdict is `baseline`, `with_skill`, or `tie`.

For every matched run pair, each judge model used must evaluate both candidate
orders. This means two pairs with one judge model require at least four judge
runs. Position swapping only the first pair is incomplete.

## Levels and claims

- `exploratory`: at least one run per side and one judge; useful for finding
  problems, not for claiming a stable effect.
- `production`: repeated matched pairs, per-pair position swaps, bound contract
  reports, declared model labels, and explicit metrics status.

`verify` proves artifact identity and protocol compliance. It does not prove
that a rubric is valid, a provider run ID is genuine, or the result generalizes
beyond the recorded tasks, models, and hosts.
