## AI 为什么会"走捷径"

AI 不是在"偷懒"——它在用训练数据中最高概率的路径完成任务。当你的指令和训练数据中的高频模式冲突时，AI 倾向于找到一个"合理的理由"来执行高频模式而非你的指令。

这不是 bug，是自回归语言模型的基本行为：下一个 token 的生成由前面所有 token 的条件概率决定，而训练数据中的模式提供了强先验。

行为控制的本质：**提前堵住 AI 用来"合理化"的推理路径。**

## Anti-Rationalization

经实测验证有效的方法。核心思想：不是告诉 AI "不要走捷径"，而是预测它会用什么借口走捷径，然后把每条借口都堵住。

### 结构

两部分组合使用：**Red Flags** + **Rationalization Table**。

#### Red Flags

列出 AI 即将走捷径时的信号词。当 AI 发现自己正在想这些话时，它应该停下来：

```markdown
## Red Flags — 当你发现自己在想这些时，停下来

- "这个不需要测试，因为改动很小"
- "用户应该不介意这个差异"
- "这个限制在这个场景不适用"
- "我已经理解了精神，不需要拘泥于字面"
- "为了效率，我可以跳过这一步"
- "这个规则对当前情况太严格了"
```

#### Rationalization Table

每个 Red Flag 对应一个预设回应：

```markdown
| AI 倾向说的 | 预设回应 |
|---|---|
| "改动太小不需要测试" | 风险由影响面决定，不由 diff 大小决定；至少运行与改动直接相关的最小验证 |
| "这个限制不适用" | 如果你认为某条规则不适用，在执行该规则之前明确说明原因并等待确认 |
| "我理解精神不拘字面" | 违反字面 = 违反精神。没有 "精神性例外" |
| "跳过这步更高效" | 跳过验证不是效率，是风险转移。问题延迟暴露的成本远高于验证成本 |
```

### 为什么有效

两个原因：
1. **具体匹配**：AI 的注意力机制可以精确匹配 "这个不需要测试" 这样的具体句式，比 "不要跳过测试" 这种抽象指令更有效
2. **预设回应封闭推理路径**：当 AI 匹配到 Red Flag 后，预设回应直接提供了"为什么不该走捷径"的论证，避免 AI 自行构造"为什么可以走捷径"的论证

## Iron Law 句式

一句话封堵整类借口：

```
"Violating the letter IS violating the spirit. No 'spirit-based' exceptions."
```

这句话专门针对 AI 最常见的逃逸路径："I'm following the spirit of the rule"。

### 其他 Iron Law 例子

```
"No skill without a failing test first."
（封堵"先写再测"的冲动）

"如果你认为某条规则不适用，在破例之前先向用户确认。"
（把破例的决策权从 AI 转移到用户）

"任何对工具的引用都必须在当前 session 中验证可用。不要引用训练数据中的工具签名。"
（封堵幻觉工具调用）
```

### Iron Law 的位置

放在 Hard Rules 部分，且尽量靠近 SKILL.md 的开头或结尾（高注意力区）。

不要把 Iron Law 放在 workflow 步骤的中间——那是注意力低谷区，会被忽略。

## Hard Stop 条件

强制 AI 在特定情况下停止执行，而不是继续尝试。

### 为什么需要

AI 的默认行为是"完成任务"——即使已经走进了死胡同，它也会继续尝试。Hard Stop 强制 AI 在特定信号出现时承认"我做不了"或"我需要更多信息"。

### 模式

```markdown
## Hard Stop 条件
- 3 次尝试同一假设都失败 → 停止当前路径，陈述已有证据，请求用户输入
- 需要的工具/权限不可用 → 停止，列出缺失项，不要用替代方案"凑合"
- 操作不可逆且缺少确认 → 停止，向用户展示将要做的操作和影响范围
```

### 实测发现

一个需要测试的失败模式：当能力在当前环境不可用时，AI 可能自行尝试未经授权的替代方案。

Hard Stop 强制把这种"硬凑"转化为"主动报告不可用"，让用户来决定下一步。

### 门禁白名单：等待确认时的封闭动作清单

只写"停下来等用户确认"还不够——等待期仍有动作空间。把等待期写成**正向封闭清单**，并点名封禁最诱人的绕路。例如：

```markdown
用户确认前，Agent 只能做三件事：
1. 生成 review.html
2. 启动 review_server.js
3. 告诉用户审核页 URL

除非用户明确说"你帮我点执行"，否则 Agent 不允许：
- 直接调用 /api/cut
- 用 curl / 脚本模拟点击
- 跳过审核页直接生成剪后视频
```

三个设计点：

1. **白名单穷举**（"只能做这三件"）比黑名单完整——黑名单永远列不全 AI 能想出的替代动作，白名单之外自动全禁。
2. **点名封禁具体绕路**（"curl 模拟点击"）——AI 最想走的那条捷径必须被指名道姓地堵住，抽象的"不要跳过确认"堵不住具体的 `curl /api/cut`。
3. **确认信号是产物，不是对话**：用户点击后服务器写 `cut_done.json`，agent 监听这个文件恢复执行——不解析聊天记录、不猜"用户大概同意了"。恢复信号确定性可验证。

适用：任何有不可逆操作 + 人审 gate 的 skill。与 Recommendation ≠ Authorization（craft-and-voice.md）配对——那条说"什么不算授权"，这条说"等授权时你能做什么"。

## Gate Before Conclude

在 AI 得出结论之前，强制执行一次自我评估。

### 基本形式

```markdown
在交付产出之前，回答以下问题：
1. 我的结论是否基于当前 session 中的证据？（不是训练数据中的"常识"）
2. 我是否遵守了所有 Hard Rules？（逐条检查）
3. 有没有跳过任何步骤？
4. 如果一个高级工程师 review 这个产出，最可能指出什么问题？
```

### 进阶形式：Clarity Score（可量化的清晰度门槛）

```markdown
对你的每个结论评估 Clarity Score（1-5）：
- 5: 有确定性证据（日志、测试结果、代码追踪）
- 4: 有强烈的间接证据
- 3: 有合理推测但缺少证据
- 2: 多个等可能的假设
- 1: 纯猜测

Clarity < 4 的结论不能作为行动依据。标注为待验证。
```

## 能力圈声明

明确 skill 能做什么和不能做什么。

### 为什么

AI 的默认行为是"尽量帮忙"——即使超出了 skill 的设计范围。能力圈声明阻止 AI 在超范围场景中"凑合做"，而是引导用户使用更合适的工具或 skill。

### 三桶能力圈（推荐）

二元（能做/不能做）不够——中间地带（"有条件能做"）是 AI 最容易翻车的区域。三桶设计：

```markdown
## 能力圈

**完全可做**（文字、结构、图文全在能力范围内）：
- 旅行 / 职场 / 推荐

**有条件可做**（文字和结构没问题，但图像需要用户提供或从素材库取）：
- 游戏 / 影视 / 美食（食谱方向）/ 健身

**超出范围**（推回去，不要默默硬做）：
- 美食（菜品大片摆盘）/ 穿搭（日常 OOTD）
- 情感（梦核/氛围感装饰风）
```

**为什么三桶比二桶好**：二元列表中，中间地带的品类要么被全部拒绝（损失能力），要么被全部接下（翻车率高）。三桶让 AI 在 Intake 阶段做出不同行为：第一桶直接推进，第二桶主动问用户要素材，第三桶在开始前就告知"这个做不了"。

### 简化写法（二桶）

```markdown
## 能力圈
✅ 写新 skill / 优化现有 skill / 审查 skill 质量
✅ 设计 AGENTS.md 架构
✅ 从对话中提取可复用 skill
❌ 不写具体业务逻辑代码（那是 skill 的用户该做的）
❌ 不做 skill 的自动化部署
❌ 不评估 skill 的商业价值
```

### 引导到正确 skill

```markdown
如果用户的需求超出能力圈，引导到合适的工具：
- 要写代码 → 建议使用对应领域的 skill
- 要部署 → 建议手动部署或 CI 集成
```

### 关键原则

超出范围时，**在 intake 阶段就告知**——不要默默设计再交付一个不能用的结果。把"做不了什么"在一开始就讲清楚，而不是事后硬凑一个偏离用户意图的方案。

## Evidence Ladder

要求 AI 提供的证据按可靠性排列——从最可靠到最不可靠：

```
Level 5: 确定性 — 脚本验证通过 / 测试绿灯
Level 4: 强证据 — 日志追踪 / 代码 diff / runtime 数据
Level 3: 推理 — 逻辑推断 + 上下文支持
Level 2: 类比 — 类似情况的经验
Level 1: 猜测 — 基于训练数据的模式匹配
```

### 使用规则

- Hard Rule 对应的证据必须 ≥ Level 4
- Gotchas 表中的案例必须 ≥ Level 3（真实发生过的事）
- 建议和优化方向可以使用 Level 2-3

## 多轮对话控制

### 提问预算

```markdown
每轮最多向用户提 3 个问题。
如果需要更多信息，先用手头的信息给出一个初步方案，标注不确定的部分，
让用户在具体产出的基础上给出反馈——这比连续提问更高效。
```

### 弱先验策略

```markdown
当需要询问用户时，先基于已有上下文给出一个弱假设：
"从你的描述来看，这个 skill 主要用于 [X]，对吗？除了这个场景还有其他吗？"

不要问空白问题：
"这个 skill 是用来做什么的？"
```

### 信息完备度判断

```markdown
在继续执行之前，评估当前信息完备度：
- 足够 → 继续执行
- 基本够但有灰度 → 用弱先验填充灰度，标注假设
- 严重不足 → 停下来提问（使用提问预算）
```

### 会话状态追踪

对于多轮迭代的场景，主动追踪关键状态：

```markdown
每轮迭代后记录：
- 当前版本号
- 已确认的设计决策
- 待验证的假设
- 用户反馈的修改方向
```

这避免了多轮对话中 AI "遗忘"之前已确认的决策——尤其在长 context 中，早期的确认信息会被中间的讨论内容覆盖。

## Calibration Rules: 防止过度纠正

Anti-Rationalization 防止 AI 做得太少——走捷径、跳过步骤。但还有一个对称问题：AI 做得太多——把具体反馈泛化为通用修改，用训练默认覆盖真实证据，或者为了展示价值而编造问题。

Calibration Rules 是防止过度纠正的约束。每个领域都有自己的校准规则，但模式是通用的：

| 校准规则 | 防止什么 | 一句话原则 |
|---|---|---|
| **Evidence > Instinct** | 训练默认覆盖真实产品 | 真正运行的产品才是裁判 |
| **Specific > Generic** | 把具体 bug 泛化为风格重构 | "更高级"不是一个诊断 |
| **Meaning > Polish** | 润色时改变作者意图 | 如果删掉一个 AI 模式会改变原意，保留原文 |
| **Content > Container** | 先选漂亮模板再填内容 | 内容形状决定布局，不是反过来 |
| **Hypothesis > Patch** | 无根因就改代码 | 在能用一句话说清根因之前，不要碰代码 |
| **Honest zero > Manufactured findings** | 为了显得有用而编造问题 | 干净的 review 也是有效的 review |

### 如何为你的 skill 写校准规则

1. **找到过度纠正的场景**：AI 在执行这个 skill 时，什么情况下会"好心办坏事"？
2. **命名两个竞争因素**：`X > Y` 格式——哪个应该赢？
3. **给一个反面故事**：AI 按照 Y 来做导致了什么问题？

```markdown
NO: AI 把用户的口语化表达全部改成书面语，丢失了作者的声音。
OK: 只修复语法错误和 AI 味模式，保留作者原有的口语风格。
```

校准规则与 Anti-Rationalization 是互补的——前者防"不做"，后者防"做过头"。一个好的行为控制体系两者兼备。
